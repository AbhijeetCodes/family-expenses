🗺️ The Big Picture
What is this app and why does it exist?

The Family Expenses app is a private web application that lets a family track their spending together — but with a twist: all the data lives in a regular Google Sheet the family already owns, not in some company's database they don't control.

📱 What you see

A fast, mobile-friendly app with charts, filters, and easy forms. It looks and feels like a professional expense app.

📊 Where data lives

A Google Sheet owned by the family. Every expense row the app shows is a row in that spreadsheet.

🔒 Who can access

Only family members whose Google email addresses the owner has explicitly approved. No strangers can sign in.

☁️ Where it's hosted

Vercel — a free cloud hosting service. Push new code and the site automatically updates within ~2 minutes.

Real-world analogy

Think of the app like a beautiful dashboard on top of a spreadsheet. Your Google Sheet is the filing cabinet; the app is the organised office that lets everyone work with those files easily — without needing to know where the cabinet is or how files are organised inside it.

Pages the family can visit

🏠

Dashboard (Home)

Monthly overview with spending totals, category breakdown, who-paid chart, and daily trend line.

📋

History (/expenses)

Full searchable list of every transaction this month, with filters and sorting.

➕

Add Expense (/add)

Quick form to record a new purchase. Takes under 30 seconds once you're familiar.

⚙️

Settings (/settings)

Manage the dropdown lists — categories, payment modes, family members, tags.

Let's break down what each layer does:

1

Google Sheets — The filing cabinet

This is where your data actually lives. The app connects to it using a "service account" — think of it as a robot employee that has permission to read and write to your spreadsheet.

2

lib/sheets.ts — The librarian

This file handles the actual communication with Google's servers. It holds the login credentials for the robot employee and knows how to ask Google Sheets for data. It's used by everything else — no other part of the app talks to Google directly.

3

lib/expenses.ts — The smart cache

After fetching data from Sheets, this layer holds a copy in memory for 60 seconds. If two family members open the app at the same time, only one request goes to Google — the second person gets the cached copy. This prevents overloading Google's API.

4

app/actions.ts — The secure gatekeeper

Every action that changes data (adding, editing, deleting) goes through this file. It re-checks that the user is logged in before doing anything. It's the last security checkpoint before data is written back to the spreadsheet.

5

components/Dashboard.tsx — What you see

This is the React component that turns raw numbers into the beautiful charts and cards on your screen. It runs in your browser and updates instantly as you change filters.

Code snippet
Plain English
// lib/expenses.ts export async function getAllExpenses() { if (cache && Date.now() < cacheExpiry) { return cache; } cache = await fetchFromSheets(); cacheExpiry = Date.now() + 60_000; return cache; }
"Give me all the expenses."

First, check if we fetched this data recently (within the last 60 seconds). If yes, return the saved copy immediately — no need to call Google.

If not, go to Google Sheets, fetch all rows, save a copy, and set a 60-second expiry timer. Next request will hit the saved copy.
The caching analogy — a restaurant's daily specials board

Imagine the kitchen writes today's specials on a board at 9am. Every waiter reads the board — they don't run to the kitchen to ask the chef every single time a customer wants to know. The board is refreshed every hour. That's caching: the "board" is the in-memory copy, refreshed every 60 seconds.

✏️ Quick Check

Why does the app cache expense data for 60 seconds instead of fetching from Google every time?

A
Google Sheets is very slow and can't handle fast requests
B
To prevent hitting Google's API rate limits and to make the app feel fast
C
Because the data changes very frequently and needs to be batched
✅ Right! Google's API has rate limits — too many requests and it'll start rejecting them. Caching means most page loads serve the in-memory copy instantly, with Google only called once per minute.
❌ Not exactly. The main reason is avoiding Google API rate limits and keeping the app fast. Most expense data doesn't change second-to-second, so a 60-second window is a good tradeoff.

📈 The Dashboard
What each card tells you

The dashboard is the home page. It shows this month's expenses through four "cards" — summary boxes — and a transaction sidebar. On desktop they're arranged in a 2×2 grid; on mobile they stack vertically.

💰

OverviewCard — The total at a glance

Shows the running total for the month. Three things live here:

  • Monthly total — sum of all filtered expenses
  • Trend pill — % change vs last month (green = less spent, red = more)
  • Month-end forecast — only visible mid-month: if you've spent ₹10,000 in 10 days, it projects ₹31,000 by month end (10,000 ÷ 10 × 31)
Forecast formula

(Total spent so far ÷ Day of month) × Days in month — simple daily average, projected forward. Only shown for the current month.

👥

PaidByCard — Who spent what

Shows horizontal progress bars, one per family member. The bar width is proportional to their share of total spending. Clicking a bar name filters the sidebar to show only that person's transactions.

A Member A ₹18,400 (62%)
B Member B ₹11,200 (38%)
🏷️

CategoryCard — Spending by type

Horizontal bars showing how much was spent in each expense category. Categories that are less than 4% of total get collapsed into a gray "Other" bucket to keep the chart readable.

There's a toggle button to switch between two badge modes:

  • % Proportion mode — what % of this month's total does each category represent?
  • Δ Comparison mode — how much more or less was spent vs last month?
📉

DailyTrendCard — Spending by day

A line chart (using Recharts) showing the total amount spent on each calendar day of the month. Click any point on the chart to filter the transaction sidebar to show only that day's expenses.

Performance note: The chart library (Recharts) is loaded lazily — it only downloads when needed, keeping the initial page load fast. While it loads, you see a skeleton placeholder so the layout doesn't jump.
📜

Transaction Sidebar — Individual expenses

On desktop, a sticky right column shows the list of individual transactions filtered by whatever you've selected. On mobile, this list appears below the cards.

Clicking a day on the daily trend chart scopes the sidebar to just that day — useful for drilling into "why was spending so high on the 15th?"

➕ Adding & Editing Expenses
What happens when you hit "Save"

The expense form is used both for adding new entries and editing existing ones. It's the same form component, just pre-filled when editing.

The form fields and how they work

FieldUI TypeSpecial behaviour
DateDate pickerDefaults to today
Expense TypeText + autocompleteFreeform — type anything new; it gets saved for next time
App / PlatformText + autocompleteSame — freeform with suggestions from Settings
Payment ModeText + autocompleteSame — freeform with suggestions
NameText inputFree description, no restrictions
CostNumber inputWhole rupees only
Paid BySelect dropdownFixed list from Settings — no freeform to prevent typos
One Time?CheckboxCheck for large irregular purchases
TagsMulti-select chipsFixed list from Settings — click to toggle
Why are Expense Type, App, and Mode freeform — but PaidBy isn't?

The three freeform fields accept new values so the family doesn't need to go to Settings every time they use a new app or category. But Paid By must be exact — if one person types "Abhijeet" and another types "Abhi", the app would show them as two different payers. Fixed dropdowns prevent that confusion.

What happens when you press "Save"

1

Form submits → shows "Saving…" spinner

The save happens on the server, so there's a brief ~300ms wait. The spinner prevents double-submits.

2

Server action re-checks login

Even if somehow someone bypassed the login screen, the server refuses to write anything without a valid session. Security in depth.

3

New categories promoted to Settings (if any)

If you typed a new Expense Type, App, or Mode that doesn't exist in Settings yet, it's automatically added to the Settings tab. Next time, it appears as an autocomplete suggestion.

4

Row appended (or updated) in Google Sheets

The data is written to the Expenses tab. Month column is auto-computed from the date.

5

Cache cleared → navigate to dashboard

The 60-second cache is invalidated immediately so the dashboard shows the new expense right away — no waiting for the cache to expire.

The "cache bust" in code
Plain English
// app/actions.ts await Promise.all([ addExpense(data), promoteLookupValues(data) ]); invalidateExpensesCache(); invalidateSettingsCache();
Do two things at the same time (in parallel for speed):
① Write the new expense row to Google Sheets
② Add any new dropdown values to Settings

Then throw away the cached data so the next page load fetches fresh numbers from Sheets.

🔍 Filters & Navigation
How the URL remembers what you were looking at

The filtering system has a clever design: all filter choices are saved in the URL. This means if you filter by "Grocery" and "UPI", the URL changes to include those choices. Sharing that URL with a family member shows them exactly what you're seeing.

Example URL:
/expenses?month=2026-05&t=Grocery,Food&m=UPI&sort=cost&dir=desc

This URL says: show May 2026 expenses, filtered to Grocery and Food categories, paid via UPI, sorted by cost descending.

What each URL parameter means

URL KeyWhat it filtersExample value
tExpense Types to excludeGrocery,Food
aApps to excludeZomato,Swiggy
mPayment modes to excludeCash
pPaid-By members to excludeB,C
gTags to excludevacation
oneHide one-time expenses?true (default)
sortSort columndate, cost, name
dirSort directionasc, desc
dateSelected day (sidebar only)2026-05-15
Why inverted logic? (Exclude vs include)

The filters work by exclusion — an empty filter means "show everything." This is intentional: when you open the app fresh, you see all your data. You hide things you don't want to see. This is more useful than "show only these" because you typically want to see most data and occasionally hide a noisy category.

Month navigation

The month is controlled separately — by the server, not filters. Arrow buttons in the header change the month URL parameter, which causes the server to fetch a different month's data from Sheets.

📱 On mobile

Tapping a day on the chart navigates to the full /expenses list page, filtered to that day. This gives more screen space than the sidebar.

🖥️ On desktop

Clicking a day on the chart filters the sidebar in place without leaving the dashboard. Both charts remain visible while browsing that day's transactions.

✅ Email is in the list

Google sends back the user's email. It matches one in ALLOWED_EMAILS. A session is created. The user is redirected to the dashboard.

❌ Email is NOT in the list

Even though Google confirmed their identity, the app rejects them. They see an error: "Access denied." The data remains private.

Adding a new family member: The owner changes the ALLOWED_EMAILS variable in Vercel's settings and triggers a redeploy. No code changes needed. Takes about 2 minutes.
The email check
Plain English
// lib/auth.ts callbacks: { signIn({ profile }) { const allowed = process.env .ALLOWED_EMAILS .split(',') .map(e => e.trim()); return allowed.includes( profile?.email ); } }
When someone signs in with Google, check their email against the approved list stored in the environment settings.

Return true = let them in.
Return false = block them, no matter what.

Two layers of protection

🛡️

Middleware (first gate)

Before any page loads, a middleware check runs. If you don't have a session, you're redirected to /login automatically. This protects all pages.

🔒

Server actions (second gate)

Every write action (add, edit, delete) re-checks your session server-side. Even if somehow you bypassed the first gate, no data modification is possible without a valid session.

1

Read current Settings tab

2

Check for duplicates (case-insensitive)

"petrol" and "Petrol" are treated as the same. Only genuinely new values are added.

3

Single batch write to Sheets

All new values are written in one API call (not one per value). Efficient and fast.

⚡ Performance Tricks
Why the app feels fast despite talking to a spreadsheet

Fetching data from Google Sheets on every click would feel slow. The app uses several techniques to stay snappy:

🗄️

In-Memory Cache (60s TTL)

After the first fetch, expense data is kept in server memory for 60 seconds. Most page navigations serve the cached copy — Google Sheets isn't called again.

Configurable via EXPENSES_CACHE_TTL_MS environment variable.

🔗

Request Coalescing

If two family members open the app simultaneously (a "cache miss" race), only one Sheets API call is made. The second person's request waits and shares the result.

⚛️

React.memo & useMemo

Dashboard cards only re-render when their specific data changes. Changing one filter doesn't redraw all 4 cards — just the ones whose numbers actually changed.

📦

Lazy-Loaded Charts

The Recharts library (~250 KB) is only downloaded when the chart is about to appear on screen. The initial page load is much lighter.

Request coalescing — the "one phone call" analogy

Imagine two assistants both need to call the same supplier at the same moment. Instead of two calls, one assistant calls and both listen. That's request coalescing — the second request "rides along" on the first's in-flight network call, then both get the answer simultaneously.

What happens after a write (cache invalidation)

When you add or edit an expense, the stale cache must be cleared so the dashboard shows fresh data. The app calls invalidateExpensesCache() after every write:

The "one rule" for understanding any file

Ask yourself: "Does this file run when I click something, or before a page even loads?"
If it runs on click or interaction → it's a browser (client) file.

Key concepts to remember

ConceptOne-line explanation
Google Sheets backendAll data lives in a spreadsheet the family owns. No private database.
Service accountA robot Google account that the app uses to read/write the spreadsheet.
TTL cacheSaves a copy of sheet data in memory for 60s to avoid constant API calls.
Cache invalidationAfter any write, the cached copy is deleted so the next load gets fresh data.
Server actionsAll data writes happen on the server (never directly in the browser) for security.
URL-backed filtersFilter choices live in the URL so they survive navigation and can be shared.
Inverted filter logicEmpty filter = show all. Non-empty = hide those values.
Auto-promotionNew freeform values typed in the expense form are silently added to Settings.
ALLOWED_EMAILSA comma-separated list of Google emails with access. Managed in Vercel settings.
Lazy loadingHeavy chart library only downloads when needed, keeping first load fast.
